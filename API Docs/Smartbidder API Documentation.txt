API Documentation

Introduction
This documentation provides guidance on how to use the Ascend Analytics SmartBidder web services to programmatically upload and download data on the operations of a node, including renewable generation forecasts and actual production, battery state of charge, generated bids, settlements, etc. The web services interface, from here on referred to as the API, is updated periodically to add new capabilities. Please verify that you are using the latest version of this document. If unsure or you encounter any difficulties that are not addressed in the Troubleshooting Guide section, reach out to your SmartBidder client representative.

Credentials
Valid credentials are required to use the API. When using the API in the interactive mode, user credentials are automatically determined from the browser and are the same as for using the SmartBidder interface. To obtain non-user client credentials for programmatic access, please contact your SmartBidder client representative.

Each API endpoint requires an Oauth (bearer) token. This token is issued by Microsoft and expires after a set period of time (minutes). To obtain the token, the application id, client id, and client secret are required. These are provided by your client representative. Client secret is set to expire every 12 months for security purposes. Do not share the secret with 3rd parties as it gives full access to your node information in the system. In addition to authentication, each connection passes through an authorization step. This verifies that the client has a sufficient role to perform the request on the specific resource and that it originates from an authorized IP address/range.

Below is a code snippet that obtains a token in Python. All modern programming languages support this functionality and there is no specific language requirement to use the API.

# credentials.py

APPLICATION_ID = "https://dataascendanalyticscom.azurewebsites.net"
CLIENT_ID = "bar"
CLIENT_SECRET = "baz"

_TENANT = "onascend.com"
_AUTHORITY_HOST_URL = "https://login.microsoftonline.com"
AUTHORITY_URL = f"{_AUTHORITY_HOST_URL}/{_TENANT}

Below is a code snippet that uses the obtained token in Python to get DA bids.

# main.py
# standard lib
from datetime import datetime
from zoneinfo import ZoneInfo

# pypi
import requests

# local
from credentials import APPLICATION_ID, AUTHORITY_URL, CLIENT_ID, CLIENT_SECRET
from msal import ConfidentialClientApplication

def get_token():
    auth = ConfidentialClientApplication(CLIENT_ID, CLIENT_SECRET, AUTHORITY_URL)
    token = auth.acquire_token_for_client([f"{APPLICATION_ID}/.default"])
    access_token = token["access_token"]
    return access_token

def make_request(access_token):
    API_BASE_URL = "https://data.ascendanalytics.com"
    url = f"{API_BASE_URL}/da-bids"
    client = "<insert-your-client-name-here>"
    iso = "ERCOT"
    resource = "<insert-your-resource-name-here>"
    tzinfo = ZoneInfo("America/Chicago")
    start_date = datetime(year=2024, month=4, day=10, tzinfo=tzinfo).isoformat()
    end_date = datetime(year=2024, month=4, day=11, tzinfo=tzinfo).isoformat()
    params = {
        "client": client,
        "iso": iso,
        "resource": resource,
        "start_date": start_date,
        "end_date": end_date
    }
    headers = {"Authorization": f"Bearer {access_token}", "Accept": "application/json"}
    response = requests.get(url=url, params=params, headers=headers)
    return response

if __name__ == "__main__":
    token = get_token()
    response = make_request(token)

Data API
Data API endpoints are listed below. All POST endpoints expect a JSON payload. At this time other payloads (XML, CSV, etc.) are not supported.

Bids
Day Ahead Bids - /da-bids GET
Used to retrieve DA bids generated by the system, only GET operation is supported. If there is no data available with the specified parameters, 204 code is returned. Invalid parameter format or missing parameters result in 400 client error code. DA bids are available for any day in the past and can be retrieved for the next market day once the scheduled optimization has run. To adjust the schedule, e.g. to generate the bids earlier or later, contact your client representative.

Parameter	Description
client	Your client name, a string. Required.
iso	The node ISO. Valid values: CAISO, ERCOT, PJM, SPP. Required.
resource	The resource UnitID, a string. Required.
start_date	The date for filtering DA bids to retrieve, in ISO8601 format, hour ending semantics. Required. Example: 2020-05-09T04:00:00.000Z
end_date	The end date for filtering DA bids to retrieve. Optional. If not provided, year-to-date is used. ISO8601 format.
return_format	Output format parameter, required. Must be set to json
strategy	Optional parameter for specifying which strategy bids to retrieve. If omitted, the bids for the active strategy are returned. Multiple strategies can be specified, separated by a comma.
include_nobids	Optional Boolean parameter. If set to true, the return includes every hour in the date range with 0 quantities in hours where no bid was generated. If omitted, defaults to false.
The output of this da-bids endpoint is a list of bid segments, each element of the list containing the following properties:

Property	Description
bid_timestamp	Timestamp when the model treats the bid as having been generated. ISO8601 format.
market	The market for which the bids were generated, da or rt.
product	The market product being offered or bought, e.g. energy, spin, etc.
segment_end	The end of the bid segment in MW. Can be negative.
segment_price	Bid/offer price, in $/MW.
segment_start	The start of the bid segment in MW. Can be negative. Valid range is from max charge power to max discharge power.
side_of_resource	Side of the resource where the bid was generated.
strategy	The strategy that generated the bid.
timestamp	The market hour for which the bid segment is being made, hour-ending semantics. ISO8601 format.
update_timestamp	Timestamp indicating when the bid was last generated or modified. In live operations, this will be close to the bid_timestamp, though it may be rounded to the nearest 10 minutes, while in backfilled data, it reflects when the backfill occurred. ISO8601 format.
override	Boolean indicating whether the bid is an override or a model generated bid.
default_bid	Boolean indicating whether the bid is a fallback/default bid. A value of true indicates that the model has not yet run or did not succeed in generating bids (this is rare).

Real Time Bids - /rt-bids GET
Used to retrieve RT bids generated by the system, only GET operation is supported. If there is no data available with the specified parameters, 204 code is returned. Invalid parameter format or missing parameters result in 400 client error code. Real time bids are generated multiple times for the same hour based on the configured planning horizon and are updated at least 1/hour. Where allowed by the ISO, RT bids can also be updated intra-hour. The update timestamp of the result indicates the time the bid was generated.

Parameter	Description
client	Your client name, a string. Required.
iso	The node ISO. Valid values: CAISO, ERCOT, PJM, SPP. Required.
resource	The resource UnitID, a string. Required.
start_date	The date for filtering DA bids to retrieve, in ISO8601 format. Required. Example: 2020-05-09T04:00:00.000Z
end_date	The end date for filtering DA bids to retrieve. Optional. If not provided, year-to-date is used. ISO8601 format.
return_format	Output format parameter, required. Must be set to json
strategy	Optional parameter for specifying which strategy bids to retrieve. If omitted, the bids for the active strategy are returned. Multiple strategies can be specified, separated by a comma.
include_nobids	Optional Boolean parameter. If set to true, the return includes every hour in the date range with 0 quantities in hours where no bid was generated. If omitted, defaults to false.
The output of this rt-bids endpoint is a list of bid segments, with the same schema as for the da-bids above.

Forecasts
/forecast-composite GET
Used to retrieve DA forecasts generated by the system, including forecasts of spikes (RT LMP > $100 and DA LMP < RT LMP). Other forecasts that are supported by the SmartBidder system may be added on request. Only GET operation is supported. If there is no data available with the specified parameters, 204 code is returned. Invalid parameter format or missing parameters result in 400 client error code.

The forecasts are provided at the 5-minute granularity and period-ending semantics. Each forecast is a probability, from 0 to 1, of the event being likely.

Parameter	Description
client	Your client name, a string. Required.
iso	The node ISO. Valid values: CAISO, ERCOT, PJM, SPP. Required.
resource	The resource UnitID, a string. Required.
start_date	The date for filtering DA forecasts to retrieve, in ISO8601 format. Required. Example: 2020-05-09T04:00:00.000Z
end_date	The end date for filtering DA forecasts to retrieve, in ISO8601 format. Required. The date semantics is inclusive-exclusive, e.g. starting at the start_date and up to but not including the end_date.
return_format	Output format parameter, required. Must be set to json
Renewable Forecasts
Used to upload (POST) and retrieve (GET) DA and RT renewable forecasts. Renewable forecasts for DA should be provided before the cutoff time, which is usually 1 hour before the bid submit deadline. The forecast must include the next 24 (or more) hours in the ISO local time zone. RT forecasts should be provided 1 hour or earlier before the applicable bid hour, depending on the ISO due time. For GET, if there is no data available with the specified parameters, 204 code is returned. Invalid parameter format or missing parameters result in 400 client error code.

/revenue GET
This endpoint is used to retrieve estimated revenue for one or more strategies, including the active strategy. Caution: the data provided is an estimate only and does not constitute settlements provided by the scheduler. Differences between estimated revenue and final settlements can occur for a variety of reasons including ISO price updates (settle price), outages and other connectivity issues that affect operations, and others. If significant differences are observed, please contact your client representative.

The revenue is provided at the 5-minute resolution and period-ending semantics, e.g. the timestamp indicates the end of the corresponding period.

Parameter	Description
client	Your client name, a string. Required.
iso	The node ISO. Valid values: ERCOT, CAISO, PJM, SPP. Required.
resource	The resource UnitID, a string. Required.
start_date	The date for data to retrieve, in ISO8601 format. Required. Example: 2020-05-09T04:00:00.000Z
end_date	The end date for data to retrieve, in ISO8601 format. Required. The endpoint uses inclusive-exclusive period-ending semantics.
return_format	Output format parameter, required. Must be set to json
strategy	Optional parameter for specifying which strategy revenues to retrieve. If omitted, the estimated revenue for the active strategy is returned. Multiple strategies can be specified, separated by a comma.
last_updated	Filter data to only include data updated since the last_updated date (ISO8601)
resolution	Change the frequency of the returned data. By default, revenue is returned as hourly. Options: 5minutely, hourly, daily, monthly, yearly
Response schema and description
The response format is a set of column headings values followed by a list of rows that satisfy the heading values. The response body could be reconstructed as a CSV.

Key	Value Description
columns	Array[string]. A list of column headers.
data	Array[Array[Any]]. A list of rows. Each row is a list of values that correspond to the row value for the given column index supplied in “columns”. For example, if the fourth string value in the “columns” array is “strategy”, the fourth value in a row array (“active”) implies that the “strategy” is “active” for that row.
The column definitions are:

Column	Description
product	String. The market product being offered or bought.
revenue	Float. The revenue gained or lost. Can be negative.
side_of_resource	String. Only applies in ERCOT and will be “na_placeholder” for all other ISOs. In ERCOT, battery bids and awards are associated with either the load side or the gen side.
strategy	String. The strategy used to calculate the revenue estimate.
timestamp	String[ISO8601 datetime-formatted]. The datetime the revenue estimate corresponds to, period ending. For example, a timestamp of 2025-01-02T00:00:00-06:00 at daily resolution corresponds for revenue obtained during various operating hours of Jan 1.
update_timestamp	String[ISO8601 datetime-formatted]. The datetime the revenue estimate was last calculated.

Battery Data
Used to upload (POST) and retrieve (GET) battery State of Charge (SOC) and availability. Power availability data is not required unless an adjustment should be made to the node max availability configuration. Power availability adjustments (derating) should be provided before the DA bid cutoff time to be taken into account during bid generation, which is usually 1 hour before the bid submit deadline. The endpoint is intended to provide real-time power status of the system so that the generated bids do not exceed the rating. For GET, if there is no data available with the specified parameters, 204 code is returned. Invalid parameter format or missing parameters result in 400 client error code.

Troubleshooting Guide
Refer here for further help.

Common Issues
Unable to obtain a bearer token
Verify the values for the client id, client token, and application id. Verify that the client secret has not expired.

Receiving a 403 response
403 response is indicating that the attempted access is unauthorized. The most common causes of this are:

Incorrect client or resource name in the request. Verify that both values are set correctly.
IP block. SmartBidder includes a software firewall by client and user for additional security. Reach out to your service representative to verify that the list of allowed IP addresses includes the one from which the connections are being attempted or to authorize the removal of IP blocking for your data.
