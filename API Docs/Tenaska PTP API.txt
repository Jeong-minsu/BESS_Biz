Introductions
Welcome to the PowerTools Platform API. This document serves as a comprehensive getting started guide for software engineers to analysts to those who just want to explore. Should you have any technical questions or encounter any issues along the way, please email customersupport@tnsk.com. Please provide any relevant info, such as the endpoint you are trying to hit, sample code/payload, and any other additional info. A member from our API team will assist you as soon as possible.

Housekeeping
The API is hosted at https://api.ptp.energy/.

There is a glossary provided at the end to define terms in more detail.

All date times are ISO 8601 and UTC unless otherwise noted.

The API implements sliding window based rate limiting per User, averaging more than 1 call per second over a small
time period will result in 429 Too Many Requests responses.

In addition to the info reflected in this document, you can see the high-level routes available at the API by
accessing the hosted Swagger page here.

Recommended Tools
These are free tools you can utilize to assist in exploring the API. The API supports Hypermedia as the Engine of Application State (HATEOAS) so having a tool that allows you to quickly parse and navigate the result is useful.

Google Chrome
We recommend using Chrome. Others can be used though your experience may differ from the intended behavior that has been built. We use a Chrome plugin to make links clickable and results pretty printed, called JSON Formatter by callumlocke.

Microsoft Edge
Now that Edge is built on Chromium, the behavior to Google's Chrome is very similar. Edge has built in pretty print features for JSON as well, but lacks the very useful feature of generating clickable links within the browser. Installing the JSON Formatter plugin is still recommended.

Postman
An intuitive, full-featured API development suite that allows you go test simple queries with hardcoded values or automate workflows with runtime variable replacement, environments, tests, and much more.

Visual Studio Code
A lightweight code editor that supports a variety of languages and frameworks. Designed to run with plugins to support the language of your choice. You can use this to rapidly write and debug code.

AI
Web requests are very common tasks and AI can help you write and debug requests. You can use a service like OpenAI's ChatGPT, Microsoft Copilot, Google Gemini, or other AI tools to help you write code in the language of your choice.

Command Line
Using a CLI alongside your fully developed solution allows for rapid prototyping or debugging. Because they're fairly low level, they allow you to fine tune parameters to understand how various changes affect the request or response.

If using Windows:

WSL - Windows Subsystem for Linux. This allows you to run a full Linux environment on Windows. You can install Ubuntu or any other distribution of your choice. CPU virtualization must be turned on to use this option. You can find instructions on how to install WSL here.

Git Bash (MINGW64) - This is a lightweight shell that comes with the Git for Windows package. It allows you to run bash commands and scripts on Windows. CPU virtualization is not required. You can download it from git-scm.com.

Alternatively you can use PowerShell; the power of .NET in a console. While powershell doesn't support bash, PowerShell aliases commands like curl and wget to Invoke-WebRequest, which is a native PowerShell cmdlet.

If using unix, just open your terminal!

Dynamic Documentation
A Commentary on Our Endpoints
In a typical API, endpoints are statically defined routes. They often map one to one with a backend controller that has bespoke logic to manage the resource. Endpoints such as GET /accounts and PUT /books/{bookId} are common patterns. These would be mapped to various HTTP Verbs such as GET, POST, PUT, and DELETE to indicate the behavior available for the resource. Each endpoint would also possibly have its own set of query string parameters, body schema, response schema, etc. 

Our APIs have this same layer of routes defined, which facilitate acts of querying and submitting data, however an additional layer exists that is dynamically driven by PowerTools schema. Various departments internal to Tenaska are able to create what is known as a “Viewport”, which houses a collection of DataPoints, and ultimately expose that through our API. This means that the API itself does not have any logic tied to market-specific endpoints such as MarketPrices or CurrentOperatingPlan, there is only a query controller which then fetches the schema and data from PTP and normalizes the response. Our API needs to respond with and receive a common payload structure, regardless of the underlying market fundamental being interacted with. 

HATEOAS
The API was developed with what is known as Hypermedia as the Engine of Application State, or HATEOAS, in mind. In essence, this defines that each response of the API should in turn inform you of what additional actions can be done from that endpoint.

Creating or Updating?
The API also allows you to create what are referred to as "dynamic" elements. These elements represent an ephemeral item, such as a transaction, bid, or offer. When an endpoint is configured to allow such behavior, the API can either update or create a new element. 

Any supplied identifier is not used during creation. PowerTools creates it’s own unique identifier during the commit process.

To update an existing element, you can supply the identifier if you have it, or the API will lookup the element by name, as there cannot be two duplicately named elements of the same type (transaction, bid, offer, etc) on the same flowday. The API will attempt to find the element on the flowdays provided (goLiveDate and expirationDate). 

A general rule of thumb for naming is to follow a template that suits your needs. Something like {Company}_VirtualBid_{SubmissionFlowday}_{InternalID}, where InternalID is an identifier generated on your side to keep track of the object, may suffice. It can be as simple or complex as you need it to be for your purposes.

Dimensionality
Dimensionality in PowerTools is a defining characteristic, or set of characteristics, of an Element or Data Point that allows users to Aggregate or filter data based on a specific characteristic.

Think of this as a dictionary of key-value pairs that can be used to filter or aggregate data. For example, a datapoint may have element dimensionality with a "Segment" with values like "1", "2", etc. This allows users to query data based on specific segments.

Elements
Element Definition
Represents the type of object. They can represent a physical thing (a corporation, a store) or a virtual thing (a sale, a purchase order). Element definitions do NOT represent a specific instance of a thing (an object), but define the nature of a thing (object). For instance, if we have an element definition called "Store," the customer will use that definition to construct what PTP's usage of a Store--and its data--will look like. Element definitions are templates for the "object(s)" they define. This parallels the concept of a class in object-oriented programming.

Element
An Element is an instance of an element definition, and is used in for object modeling. Elements represent actual business assets, entities, and processes that are related to each other within the boundaries of valid element definition relationships. Represents an instance of an Element Definition. User Generator A is a Generator for example. This parallels the concept of an object in object-oriented programming.

Element Dimensionality
Refers to the dimensionality that can be defined at an element definition and is applied to an element.

Query String Data Type Reference
Array[T]
The API supports two methods of passing collections. 

Using a semi-colon (;) as a delimiter for using multiple values. This was done to allow element and datapoint names to include special characters such as , or :. 

Example: {path}/query?begin=2025-01-01&dataPoints=DA_LMP;RT_LMP;Status

Using repeated query string parameters

Example: {path}/query?begin=2025-01-01&dataPoints=DA_LMP&dataPoints=RT_LMP&dataPoints=Status
